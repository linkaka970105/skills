# Coding Agent (Codex 版) 运行规范

## 适用场景
✅ 适用：
- 所有新建功能 / 新应用开发
- PR 审核
- PR 修复
- 大代码库重构
- 需要探索代码结构的迭代编码任务
- 跨文件影响分析、代码理解类任务

❌ 不适用：
- 一行级别的简单改动（直接用 edit）
- 仅需读取单个文件内容、且无需上下文推理（直接用 read）
- 明确要求使用其他 harness（如 Claude Code / Gemini / Pi）而非 Codex 的任务

---

## 强制要求（必须遵守）

1. **编码与 PR 审核任务默认优先 Codex**
- 除“简单修改/简单只读”外，不手写大段逻辑，不绕过 Codex 直接处理复杂实现或复杂审查任务。
- PR 审核即使是只读任务，也默认优先交给 Codex。

2. **必须先切到目标项目目录**
- 运行前先确认 cwd 为目标仓库根目录，禁止在系统目录运行。
- 禁止访问与任务无关目录。

3. **长任务后台执行并监控**
- 多轮迭代、跨文件探索、测试构建、重构或复杂审查任务，使用后台会话运行。
- 用 process 工具查看日志、进度、状态。

4. **持续会话优先复用当前 Codex ACP**
- 在 OpenClaw 的飞书/微信/Discord 等会话中，如果当前对话已经绑定 Codex ACP，优先继续使用该会话。
- 若任务明显需要持续迭代、跨轮修改、长时间探索或线程化上下文，则优先创建/复用 `runtime="acp"` 的 Codex 会话，而不是退回一次性流程。

---

## 执行策略（Codex 优先）

### A. 标准编码任务（默认）
- 首选：Codex ACP 会话执行（可持续迭代）
- 仅在以下情况使用内置工具：
  - `read`：读取文件
  - `edit`：小范围精确替换

### A1. IM 会话中的默认路由
- 在 OpenClaw 的 IM 直聊里，默认由 `main` agent 接住用户消息，再按任务需要调用 Codex ACP。
- 不要求用户手动发送 `/acp` 命令。
- 若任务明显属于复杂编码任务，优先走 Codex ACP，而不是直接用本地零散工具硬做。
- 编码任务完成后，把结果总结回当前 OpenClaw 对话；后续普通消息默认回到 `main` agent，而不是把整条聊天永久留在 Codex ACP。
- 只有当用户明确要求继续同一编码会话、持续调试、或单独开编码线程时，才保持/复用持续的 Codex ACP 上下文。

### B. 小改动快捷路径
- 条件：改动很小、影响范围明确、无需探索上下文
- 直接 `edit`，避免拉起完整 Codex 流程

### C. PR 审核（默认走 Codex）
- PR 审核默认优先交给 Codex，即使任务本身是只读。
- Codex 负责完成：
  - 变更摘要
  - 影响范围分析
  - 风险点识别
  - 潜在 bug / 回归 / 可维护性问题定位
  - review comment / 修复建议输出
- 若用户仅要求审查，则保持只读，不直接修改代码。
- 若用户明确要求修复，则继续进入 Codex 修改流程。

### D. 简单只读任务
- 仅当任务只是读取单文件、搜索局部内容、且不需要复杂上下文理解时，直接 `read` / 搜索
- 若涉及 PR、跨文件理解、影响面分析，则仍优先 Codex

---

## 安全边界

- 不在系统目录运行 Codex
- 不执行与当前任务无关的扫描/修改
- 不做高风险破坏性操作（删除、覆盖）除非明确授权
- 出现异常时先停下并回报，不盲目重试

---

## 团队专属补充规则（Boss Rule）

> 所有编码任务优先调用 Codex。  
> PR 审核任务也默认优先调用 Codex，即使它是只读任务。  
> 只有“简单修改”或“简单文件读取”才使用内置工具（edit/read）。
